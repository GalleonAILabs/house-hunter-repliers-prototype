## Problem Statement

House Hunter's alpha rebuild is generalizing from a 2-person hardcoded model
(Mark/Katie) to a dynamic N-person buyer_group/realtor_team model, on top of
a Repliers API integration, ahead of a concrete demo: Anees and his partner
using the app alongside Mark and Katie. The app currently has no persistence
layer at all (stdlib Python server, static JSON, localStorage only for UI
prefs), no working map on mobile, and no data source that returns real
Ontario listings. All three of these have to be resolved together before the
demo, not sequenced as separate phases.

## Premises Challenged and Confirmed

1. **Repliers proves nothing for Ontario today.** Free tier returns zero
   Canadian listings (`state=ON` and Toronto filters returned 0 as of
   2026-07-02). Confirmed. Fix: swap the sample query to ~300 listings from
   one major US metro to prove the integration pipeline at realistic volume
   while paid PropTx/ITSO access is pending. This is not abandoning
   Repliers, it is proving the pipeline honestly with data it can actually
   return today.

2. **The N-person data model is not speculative.** Challenged as "built
   ahead of a validated second user." Rejected: Anees is actively reviewing
   the project, and the demo goal is Anees plus his partner using the app
   alongside Mark and Katie. Four named, real people. Building the dynamic
   model now is directly required for that demo, not a bet on a future
   customer.

3. **There is a real architecture gap: no persistence layer exists.**
   Confirmed. The data model in `DATA_MODEL_NOTES.md` assumes a relational
   store; the codebase has none. Resolution: SQLite (lightweight, no server
   process, stdlib-adjacent via Python's `sqlite3`, sufficient for 4 known
   users). The actor selector and the SQLite layer are one unit of work, not
   sequenced phases.

4. **Map and List are not sequenced by priority, they are one UI.**
   Challenged as "actor selector before map, driven by what felt tractable."
   Rejected: Map (pins, filters, card popups) and List (cards, same filters)
   are equal-priority twin views, both carrying the "Who Am I" actor
   selector. The Leaflet break on Android/localtunnel is a tooling problem
   (phone-over-tunnel testing), not a signal to deprioritize the map.
   Resolution: build actor selector, persistence, and map together; fix
   Leaflet by testing in a proper dev environment instead of a phone tunnel.

   **Reconciling with the prior parking decision.** `HANDOFF.md` (2026-07-02)
   and `PROJECT_BRIEF.md` (2026-07-03) explicitly parked the map and told
   Claude Code not to spend further time on it, after seven commits of
   layout/timing fixes (flexbox, `min-height:0`, ResizeObserver, double rAF,
   explicit pixel height, literal POC-pattern copy) failed and the written
   conclusion was that the bug "appears specific to Leaflet tile rendering
   in this Android Chrome/localtunnel context," not a normal layout bug.
   This plan reopens that decision on a narrower basis: none of those seven
   attempts changed the test harness itself (phone over a localtunnel
   URL). The untested variable is the tunnel/mobile-browser combination,
   not another CSS or JS timing fix. This is worth one time-boxed attempt
   because it targets a variable the prior seven attempts did not touch,
   but if it fails, this reopens the exact same unresolved bug HANDOFF.md
   already gave up on, so it needs a hard time box and a named fallback
   (see Next Steps item 4 and its fallback).

## Engineering Review Decisions (locked, /plan-eng-review 2026-07-04)

The office-hours design above was reviewed section by section (architecture,
code quality, tests, performance) plus an independent outside-voice pass. The
outside voice caught two severe gaps the architecture review missed by
staying backend-only: no plan existed for the actual interactive feedback UI
(it doesn't exist yet in `static/app.js`), and no plan existed to preserve
the 104 rows of real Mark/Katie feedback already recorded in
`data/poc_listings.json`. Both are now locked below alongside the backend
decisions.

- **D1 (concurrency):** `server.py` already runs `ThreadingHTTPServer`
  (server.py:397). Each request opens its own short-lived `sqlite3`
  connection (WAL mode) rather than sharing one connection across threads,
  standard, avoids `check_same_thread=False` footguns.
- **D2 (latest state):** `listing_feedback` stays an append-only event log.
  "Current state" (latest rating/status per person per listing) is derived
  on read via SQL (latest row per `person_id` + `action_type`), not a
  second table that could drift out of sync.
- **D3 / D11 (auth):** A shared-secret header token (env var, same pattern
  as the existing Repliers key) is required on **both** `GET` and `POST`
  endpoints, not POST-only. Documented explicitly as a bot/drive-by
  deterrent for the public localtunnel URL, **not real access control**.
  The token lives in browser JS and is visible in dev tools, same exposure
  class this project already bans for the Repliers key. Anyone who has the
  token can still read every private note and reject-reason.
- **D4 / D10 (validation):** `POST /api/feedback` validates `person_id`
  against the `people` table and `listing_id` against an in-memory set of
  known POC listing ids (`POC-{row}`), loaded at server startup from
  `data/poc_listings.json`. Repliers-sourced ids (`mlsNumber`) are
  format-checked only (non-empty string), not existence-checked. Repliers
  is a live external source with no canonical id list this server owns, and
  checking existence would mean a live network call on every write.
  Invalid `person_id`/`listing_id` returns 400 with a clear error code,
  never a generic 500 or a silent orphan row.
- **D5 (tests):** Add `test_server.py` using Python's stdlib `unittest`
  (no new dependency) against an in-memory SQLite db. Covers schema
  creation, feedback insert/validation, and the latest-state query. See the
  Test Plan artifact for the full interaction/edge-case list.
- **D6 (batch reads):** `GET /api/listings/:id/feedback` (per-listing) is
  replaced by `GET /api/feedback?listing_ids=...` (batch). Rendering
  ~300-400 listings on Map or List needs feedback state for all of them at
  once; a per-listing endpoint would mean 300+ requests on page load.
- **D7 (UI build scope, from outside voice):** `static/app.js` has **no
  interactive feedback UI today**. `populateCard()` only displays
  `markRank`/`katieRank`/comments as static read-only text. Building the
  rate control, note composer, reject-with-reason flow, and research-request
  control (for both List cards and, later, Map popups) is real, uncosted
  work beyond Approach A's original "Effort: M" backend estimate. Sized as
  its own Next Steps item, likely the single largest piece of work in this
  plan.
- **D8 (POC data migration, from outside voice):** The 104 existing
  Mark/Katie ratings and comments in `data/poc_listings.json` are backfilled
  into `listing_feedback` as a one-time migration, attributed to Mark's and
  Katie's `people` rows. Without this, the demo would show their real,
  already-recorded opinions as blank the moment the new feedback store goes
  live, a visible regression in front of Anees on the exact data that
  proves the product's value.
- **D9 (dynamic card rendering, from outside voice):** `populateCard()`
  hardcodes `item.markRank`, `item.katieRank`, and the literal strings
  `"Mark"`/`"Katie"`/`"Anees"`, exactly the 2-person hardcoding this whole
  redesign exists to eliminate, sitting in the frontend rather than the
  schema. Rewritten to loop over `GET /api/people` and render each person's
  latest feedback (from the D6 batch endpoint) dynamically.
- **D12 (sequencing, supersedes Premise 4's original stance):** Build one
  shared feedback-UI component (rating/note/reject/research controls, batch
  feedback fetch) consumed by both List cards and Map popups. Ship the
  List-view wiring first, since it's guaranteed to demo regardless of
  Leaflet's outcome. Defer Map-pin-specific wiring (colored pins, popup
  rating controls) until after the Leaflet spike (Next Steps item 6)
  resolves one way or the other. This reverses Premise 4's original "build
  actor selector, persistence, and map together" sequencing after the
  outside voice flagged that building map-pin-specific work before a
  known-risky, previously-abandoned bug is resolved risks wasting that work
  if the hard-stop fires. The shared-component approach keeps as much of
  Premise 4's "one coherent UI" intent as possible while addressing the
  actual risk (wasted work), not overriding the premise wholesale.
- **D13 (demo logistics):** Confirmed in-person, same Wi-Fi. The local
  network Leaflet test (Next Steps item 6) doubles as validating the actual
  demo setup, not just a diagnostic. No separate tunnel-specific fix is
  needed on top of it.

## Approaches Considered

### Approach A: Minimal viable, SQLite bolt-on (chosen)
- Add `sqlite3` (Python stdlib, no new dependency), opened with WAL mode.
  Four people on one demo device is low write volume; WAL mode avoids
  writer-blocks-reader stalls without needing a queue or app-level locking.
- Minimal tables sized to the actual demo: `people` (id, name, role:
  buyer/advisor, `buyer_group_id` stub column, unused for now but present
  so Approach B's real `buyer_groups` table can be added later without a
  column migration on `people`), `listing_feedback` (person_id, listing_id,
  action_type, rating, status, note, reason, created_at) as an append-only
  event log.
- Minimal API surface on the existing stdlib server: `GET /api/people`
  (list selectable actors), `POST /api/feedback` (body: person_id,
  listing_id, action_type, plus the relevant field), `GET
  /api/listings/:id/feedback` (all feedback for a listing, for map popups
  and cards to render per-person state). Exact request/response shapes are
  deferred to `/plan-eng-review`, not because they're unimportant, but
  because they're a schema-lock decision, not a build-order decision.
- Actor selector: dropdown in the top bar (visible on both Map and List
  views), backed by `GET /api/people`. Selecting a person sets the active
  actor for all subsequent rating/note/reject/research actions; persisted
  per-device via localStorage (same pattern as the existing dark-mode
  toggle) so a shared household device doesn't reset to "no one selected"
  on every reload. Switching actor does not clear or reload listing data,
  only changes attribution going forward.
- Map: per the "Map identity rule" in `DATA_MODEL_NOTES.md`, pins and card
  popups must reflect the active actor's own ratings/notes/status, not a
  blended view. This applies once Leaflet itself renders. It is in scope
  for this approach, not a follow-on.
- Map rendering fix: see Next Steps item 4 for the time-boxed procedure and
  fallback. Risk on this specific item is Medium-High, not Low (see
  Feasibility note below); it's called out separately from the rest of
  Approach A, which is genuinely low-risk.
- Repliers: swap the sample query to ~300 listings from one major US metro
  to prove pipeline volume and realism while Canadian access is pending.
  Verify the free tier actually returns that volume in one call before
  committing to the number (see Next Steps item 5).
- Effort: M. Risk: Low for SQLite/actor-selector/Repliers-swap; Medium-High
  specifically for the Leaflet fix (see above and Feasibility).
- Reuses: existing `server.py`, `static/app.js`, current card/list UI,
  existing localStorage settings pattern.
- **Deviation from `DATA_MODEL_NOTES.md` acknowledged.** That doc says to
  "store buyer groups and buyer group members properly from the start" to
  avoid rebuilding when a search has 1, 2, or 5 decision-makers. A flat
  `people` table without a real `buyer_groups` table is a narrower version
  of that. The principle it actually protects, never hardcode a fixed
  count of people, never assume 2, is honored: `people` is queried
  dynamically with no assumed roster size. What's deferred is only the
  *second* table (grouping people into named groups), which has exactly one
  real instance today (this one household plus Anees's team for the demo).
  Adding a real `buyer_groups` table before a second group exists is
  Approach B's mistake repeated in miniature. The `buyer_group_id` stub
  keeps that upgrade a column-population step, not a rebuild.

### Approach B: Ideal architecture, full relational schema now
- Implement the complete schema from `DATA_MODEL_NOTES.md` up front:
  `buyer_groups`, `buyer_group_members`, `realtor_teams`,
  `realtor_team_members`, `buyer_group_realtor_team_assignments`,
  `listing_feedback` as an event log.
- Seed script creates real rows for Mark, Katie, Anees, and Anees's partner.
- Actor selector reads dynamically from both member tables with role labels
  (buyer vs. advisor).
- Effort: L/XL. Risk: Medium, burns days on hierarchy tables (multi-team
  assignments) with zero real usage yet, risking the demo date.
- Reuses: `DATA_MODEL_NOTES.md` schema directly, existing Repliers proxy.

### Approach C: Creative, JSONL event log first, SQLite later
- Actor selection via localStorage per device, reusing the existing
  dark-mode/settings-drawer pattern.
- Every feedback event POSTs to a small `/api/log-event` endpoint that
  appends one JSON line per event (person-attributed, never overwritten).
- Migrate JSONL to SQLite after the demo once real usage patterns are known.
- Effort: S/M. Risk: Low short-term, Medium later, guarantees a migration
  task right after the demo, and blocks SQL-based consensus filtering
  (per-person/group filters) that's needed soon after.
- Reuses: existing localStorage settings-drawer pattern.

## Recommended Approach

**Approach A.** It satisfies every confirmed premise (real persistence,
actor selector and map built together, Repliers sample swap) sized exactly
to the four known demo participants, without pre-building org-hierarchy
tables nobody has used yet. It upgrades cleanly to Approach B's schema if
and when a second realtor team actually shows up.

## Failure Modes

| Codepath | Realistic failure | Test? | Error handling? | User-visible? |
|---|---|---|---|---|
| Concurrent writes (two people rate near-simultaneously) | WAL-mode write contention | [→E2E], not unit-testable | Handled at DB level (D1/WAL) | No, transparent |
| `POST /api/feedback` invalid `person_id`/`listing_id` | Stale localStorage after a demo reset | Yes (D4/D10 test plan) | 400 with clear code (D4) | Yes, clear error |
| Backfill migration (D8) runs on repeated server restart | Duplicate Mark/Katie feedback rows | Not directly tested | Guarded by D14 (idempotency check) | Would be silent without D14. **This is why D14 was added as a critical gap, now closed** |
| Missing/invalid auth token (D3/D11) | Demo device's token misconfigured | Not directly tested | 401 | Yes, but needs a clear-enough UI message. **Flag for implementation, not currently specified how the frontend surfaces a 401** |
| Malformed JSON body on `POST /api/feedback` | Client bug sends bad payload | Yes (D4 test plan) | Falls through to server.py's existing generic exception handler (server.py:392-393) | Yes, though as a raw exception name/detail, not a friendly message |

No critical gaps remain **unaddressed** after D14; the one residual note
(auth-failure UI messaging) is minor and flagged for implementation
attention rather than blocking.
